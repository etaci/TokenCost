# Fusebox · 技术栈与各组件技术路线

> 配套文档：`架构.md`
> 版本：v1.0 · 2026-05-03
> 原则：**单 binary 部署 / 性能优先 / 主流生态 / 故事化营销友好**

---

## 0. 总体技术决策原则（PM 立的「红线」）

| # | 原则 | 含义 | 为什么 |
|---|---|---|---|
| 1 | **关键路径用 Rust** | Proxy / Policy Engine 用 Rust | 性能（p99 <5ms）+ 单 binary + Rust 在 GitHub 社区"性感"，吸 star |
| 2 | **外围用 TypeScript** | Dashboard / SDK / MCP / CLI 周边用 TS | 生态最大、贡献者最容易招、UI 库最丰富 |
| 3 | **Python 只做 SDK** | 不写后端 Python | Python 后端运行 overhead 大、与 Indie 单 binary 哲学冲突 |
| 4 | **不依赖 SaaS** | 测试 / CI / 部署不绑定特定云 | Phase 1-2 完全本地跑；自托管用户能直接用 |
| 5 | **可插拔** | DB / 队列 / cache 都走 trait/interface | Phase 4 升级 ClickHouse / Kafka 不用重写 |
| 6 | **零配置工作** | 不写任何 config 也能跑 | DX 是爆款生死线 |

## 3. 各组件技术路线（深潜）

### 3.1 Fusebox Proxy（Rust）

**职责**：OpenAI / Anthropic / Bedrock / Google / OpenRouter 兼容的 HTTP 网关，所有调用先经过这里。

**技术选型**：
| 项 | 选择 | 备选 | 决策依据 |
|---|---|---|---|
| Web framework | **Axum 0.7** | actix-web、warp、rocket | Tokio 原生、Tower 中间件生态、tower-http 直接复用 |
| HTTP client | **reqwest** | hyper（裸）、ureq | reqwest 上手快；性能差距对 LLM IO 场景可忽略 |
| 异步运行时 | **Tokio** | async-std | 事实标准 |
| 序列化 | **serde + serde_json** | sonic-rs | sonic-rs 更快但生态弱；MVP 用 serde |
| Token 计数 | **tiktoken-rs** + **tokenizers** (HF) | gpt-tokenizer | OpenAI 用 tiktoken-rs、Anthropic 用 tokenizers，必须同时支持 |
| 流式 | SSE pass-through | 自己解析重打包 | pass-through 几乎 0 overhead |
| 配置 | **figment** + YAML/TOML/env 多源 | config-rs | figment 错误信息友好 |
| 日志 | **tracing** + tracing-subscriber | log + env_logger | tracing 是 async 时代标配 |
| 指标 | **metrics** crate + prometheus exporter | tracing-opentelemetry | prometheus 端点是社区公认 |
| 错误处理 | **thiserror** (lib) + **anyhow** (bin) | snafu、eyre | 主流二选一 |

**关键代码骨架**（伪代码示意）：
```rust
async fn handle_chat(
    State(ctx): State<AppCtx>,
    headers: HeaderMap,
    body: Bytes,
) -> Result<Response, FuseboxError> {
    let tenant = identify(&headers, &ctx)?;

    // 1. Pre-flight policy check
    let request = parse_request(&body)?;
    let estimate = estimate_cost(&request, &ctx.pricing)?;
    let decision = ctx.policy.evaluate(&tenant, &estimate).await?;

    match decision {
        Decision::Allow => {/* fall through */},
        Decision::Deny(reason) => return Ok(deny_response(reason)),
        Decision::Downgrade(model) => {/* swap model */},
    }

    // 2. Forward
    let upstream_resp = ctx.upstream.forward(&request).await?;

    // 3. Post-process & ledger
    tokio::spawn(async move {
        let actual_cost = extract_cost(&upstream_resp, &ctx.pricing);
        ctx.ledger.record(tenant, actual_cost).await;
        ctx.policy.post_evaluate(&tenant).await;
    });

    Ok(upstream_resp.into())
}
```

**性能目标**：
- p50 latency overhead ≤ 1ms
- p99 latency overhead ≤ 5ms
- 单实例 10k req/s（小 payload）
- 内存常驻 < 100MB

**Bench 工具**：`criterion.rs` 单元 bench + `wrk2` / `oha` 端到端压测，CI 卡点。

### 3.2 Policy Engine

**职责**：评估"允许 / 拒绝 / 降级"，是产品的灵魂。

**两层结构**：
1. **Rule Layer（声明式）**：用户在 YAML 写规则
2. **Script Layer（命令式）**：高级用户写 Rhai 脚本（沙箱化）

**Rule YAML 示例**（用户写的）：
```yaml
policies:
  - name: indie-default
    match: { tenant: "*" }
    rules:
      - type: budget
        window: 1d
        limit: $10
        on_exceed: deny

      - type: rate
        window: 1m
        limit: 60
        on_exceed: queue

      - type: anomaly
        metric: spend_rate
        method: ewma
        sigma: 3
        on_trigger: trip_breaker

      - type: cost_per_request
        limit: $0.50
        on_exceed: downgrade
        downgrade_to: gpt-4o-mini
```

**异常检测算法**：
- **EWMA + 3-sigma**（MVP）：指数加权移动平均，简单、有效、无需训练
- 后期：HOT-SAX / Twitter AnomalyDetection 端口、孤立森林（用 `linfa`）

**断路器规则**：
```rust
enum BreakerState { Closed, Open { until: Instant }, HalfOpen { trial_count: u32 } }

impl BreakerSM {
    fn trip(&mut self, reason: TripReason) { /* Closed/HalfOpen → Open */ }
    fn try_close(&mut self) { /* Half-Open → Closed iff success rate ≥ 95% */ }
}
```

### 3.3 Cost Ledger

**职责**：持久化所有 spend 事件，支持时序查询、聚合、按 tenant 切分。

**数据模型**（Postgres + TimescaleDB）：
```sql
-- 主事件表（hypertable）
CREATE TABLE spend_events (
    ts          TIMESTAMPTZ NOT NULL,
    tenant_id   TEXT NOT NULL,
    project_id  TEXT,
    user_id     TEXT,
    provider    TEXT NOT NULL,            -- openai / anthropic / ...
    model       TEXT NOT NULL,
    input_tokens INT NOT NULL,
    output_tokens INT NOT NULL,
    cost_usd    NUMERIC(12,6) NOT NULL,
    request_id  UUID,
    metadata    JSONB
);
SELECT create_hypertable('spend_events', 'ts');

-- 连续聚合（5min/1h/1d 滚动）
CREATE MATERIALIZED VIEW spend_5m WITH (timescaledb.continuous) AS
SELECT time_bucket('5 minutes', ts) bucket, tenant_id, model,
       SUM(cost_usd) cost, SUM(input_tokens + output_tokens) tokens
FROM spend_events GROUP BY bucket, tenant_id, model;
```

**实时计数器**：Redis HINCRBYFLOAT
```
key:    fb:spend:{tenant}:{window}
value:  {model: cumulative_cost_usd}
TTL:    window 大小 + 30s buffer
```

**为什么不用 ClickHouse**：
- ClickHouse 运维复杂、不适合 OLTP（断路器需要写 + 立即读）
- TimescaleDB 一个 DB 同时搞定 OLTP + 时序，对自托管用户极友好
- Phase 4 真的要做 100B+ events/月，再上 ClickHouse

### 3.9 Pricing 数据管理

**问题**：各家 LLM 定价频繁变化（Anthropic 一年改 3 次，新模型每月有），定价错了 = 用户赔钱 = 信任崩塌。

**方案**：
1. **数据源**：fork LiteLLM 的 `model_prices_and_context_window.json`（社区维护、最权威）
2. **存储**：仓库内 `pricing/*.yaml`（人类可读）
3. **同步**：GitHub Actions cron（每天）→ 拉 LiteLLM JSON → diff → 自动 PR
4. **嵌入**：编译时 `include_str!()` 进 binary，**离线可用**
5. **覆盖**：用户可以在 config 里 override 某个模型价格

**示例 `pricing/anthropic.yaml`**：
```yaml
provider: anthropic
last_updated: 2026-04-15
models:
  claude-sonnet-4-5:
    input_per_1m: 3.00
    output_per_1m: 15.00
    cache_read_per_1m: 0.30
    cache_write_per_1m: 3.75
  claude-opus-4:
    input_per_1m: 15.00
    output_per_1m: 75.00
```

## 4. 关键技术决策（X over Y）—— 用面试题方式呈现

### Q1：为什么 Proxy 是 Rust 而不是 Go？
- **性能**：Rust 在 IO-bound 场景与 Go 接近，但内存占用低 30-50%
- **单 binary 体积**：Rust 编译后 ~15MB（开 LTO + strip），Go ~25MB
- **社区营销**：GitHub 上 Rust 项目天然吸 star（"Built in Rust" 是个 meme）
- **错误处理**：Rust 的 `Result` + `?` 在网关场景比 Go 的 `if err != nil` 更清爽
- **唯一让步**：Rust 招人难度 > Go；但我们靠 OSS 社区 contributor，不靠雇佣

### Q2：为什么 Postgres + TimescaleDB 不是 ClickHouse？
- **运维**：自托管用户能直接用现有的 Postgres
- **OLTP + 时序合一**：断路器需要写完立即读（强一致），ClickHouse 是异步 merge
- **TimescaleDB Apache-2.0** 版本免费、够用
- 升级路径：到了 100B 级才上 ClickHouse 做 sink

### Q3：为什么 Dashboard 用 Next.js 不是 SvelteKit / SolidStart？
- **生态**：Tremor / Shadcn 是 React-only，重做要 200+ 小时
- **贡献者**：Next.js 工程师存量是 Svelte 的 10 倍
- **RSC**：Server Components 让 dashboard "感觉" 极快
- **唯一让步**：Bundle 比 Svelte 大；我们用 RSC 抵消

### Q4：为什么用 Better-Auth 不是 Clerk / Auth0？
- **零依赖外部 SaaS**：自托管用户不能要求他们注册 Clerk
- **开源、Headless**：UI 自己画
- **2026 主流化**：Better-Auth 已是 OSS 圈的 Next.js Auth 替代

### Q5：为什么 MCP Server 用 TypeScript 不是 Rust？
- Anthropic 官方 SDK TypeScript 版最完善
- MCP 协议本身轻量，不需要极致性能
- 与 dashboard 共享类型定义（`shared/types.ts`）

### Q6：为什么不直接用 LiteLLM 做底层 / 提交 PR 给 LiteLLM？
- **战略**：我们要自己的品牌、自己的 narrative。"a feature in litellm" vs "Fusebox" 的传播力差 10x
- **战术**：复用其 pricing 数据、与 LiteLLM 兼容、做生态而非死磕

### Q7：为什么不做 SaaS-only？
- 看 Sentry / PostHog / Plausible — 开源 + 自托管才能爆款
- Indie hacker / 开发者更信任开源
- 自托管 = 数据主权，企业关注

## 6. 测试策略

### 6.1 测试金字塔
```
        ┌─────────────┐
        │   E2E (5%)  │  docker-compose + playwright
        ├─────────────┤
        │ Integration │  cargo test --features it
        │    (20%)    │  vitest with msw
        ├─────────────┤
        │  Unit (75%) │  cargo test / vitest / pytest
        └─────────────┘
```

### 6.2 关键测试场景
1. **断路器状态机**：所有迁移路径、并发触发不能抖动
2. **Token 计数准确性**：与 OpenAI / Anthropic 实际返回的 usage 字段对比，误差 < 1%
3. **Pricing 计算**：所有模型 fixture 对比已知正确值
4. **流式 pass-through**：分块到达 / 网络中断 / 客户端断开
5. **多 tenant 隔离**：A 租户的 spend 不影响 B 租户的 breaker
6. **Anomaly detection 正例 / 反例**：合成数据集验证误报率

### 6.3 Chaos / Fuzz
- `cargo fuzz` on JSON parser
- 注入 upstream 延迟 / 错误 / 部分响应
- Redis / Postgres 不可用时的 fallback

## 7. 部署技术路线

### 7.1 部署形态（按用户友好度排序）
| 形态 | 命令 | 目标用户 |
|---|---|---|
| **One-line install** | `curl -fsSL fusebox.dev/install.sh \| sh` | Indie hacker |
| **Docker Compose** | `docker compose up` | 大多数自托管 |
| **Single binary** | `cargo install fusebox` / `brew install fusebox` | Rust / Mac 用户 |
| **Helm chart** | `helm install fusebox fusebox/fusebox` | Enterprise / K8s |
| **systemd unit** | 提供 `.service` 文件 | Linux 服务器 |

### 7.2 默认配置（零配置工作）
```yaml
# 用户什么都不写时的隐式默认
fusebox:
  bind: 0.0.0.0:8080
  upstream:
    openai: { url: https://api.openai.com }
    anthropic: { url: https://api.anthropic.com }
  storage:
    type: sqlite               # 没配 PG 就用 SQLite，零依赖
    path: ~/.fusebox/data.db
  cache:
    type: in-memory            # 没配 Redis 就内存
  policy:
    default_budget_per_day: $50  # 兜底
```

**SQLite fallback 是关键**：让用户能 30 秒上手，不用先配 Postgres。

## 8. 安全与合规

### 8.1 安全设计
| 项 | 方案 |
|---|---|
| API key 存储 | **不存**，pass-through 到 upstream（避免成为攻击面） |
| TLS | proxy 默认 listen HTTP，TLS 让 nginx/caddy 终结；可选 rustls 内置 |
| 内部认证 | Bearer token (HS256 JWT) for SDK ↔ proxy |
| Audit log | 所有策略决策、breaker trip 写到 audit table |
| Secrets in config | `${ENV_VAR}` 占位符 + 不打印 |
| Supply chain | `cargo audit` + `npm audit` 卡 CI |
| SBOM | `cargo cyclonedx` + `npm sbom` 每个 release 附带 |

### 8.2 合规友好（为 Phase 4 企业版铺路）
- 全量 audit log
- Data retention 可配置（30/90/365 天）
- 明确不收集任何 prompt / completion 内容（只收 metadata）
- GDPR 友好（per-user delete）
- 后期 SOC2 Type 1（外包审计公司，预算 ~$15k）

## 9. 我们自己用什么观测自己（吃狗粮）

| 用途 | 工具 |
|---|---|
| Logs | tracing → tracing-subscriber → JSON stdout → docker logs |
| Metrics | metrics crate → Prometheus exporter `/metrics` |
| Traces | tracing-opentelemetry → OTLP → 任何 backend |
| Errors | Sentry self-hosted（Phase 2 起） |
| Anonymous telemetry | 我们自建 endpoint（可 opt-out，类似 VSCode）收集：版本、OS、active tenant 数（仅数字） |

**重要**：telemetry 必须 **opt-out 而非 opt-in**，但要在第一次启动时 banner 明示，符合社区伦理。

- 复杂的 RBAC — 仅做 admin / member 二级
- 跨 region 数据复制
- Mobile app（先做响应式 web 即可）
